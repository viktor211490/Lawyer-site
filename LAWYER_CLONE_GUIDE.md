# Дубль проекта под нужды адвоката (отдельный репозиторий)

Цель: быстро создать отдельный репозиторий на базе текущего Psycho-site, при этом **не тащить** артефакты сборки и локальные базы данных, и заранее определить, что нужно менять/добавлять **только если это действительно требуется**.

## 1) Создание репозитория-дубля

В репозитории добавлен скрипт:

- `scripts/create-lawyer-clone.ps1`

Он копирует проект в новую папку, исключая `.git`, `node_modules`, `bin/obj`, `dist`, локальную SQLite и её WAL/SHM файлы, затем инициализирует новый git‑репозиторий.

Пример:

```powershell
pwsh .\scripts\create-lawyer-clone.ps1 -DestinationPath "C:\Users\alexe\repos" -RepoName "Lawyer-site"
```

## 2) Минимальный набор правок после клонирования (обязательное)

### Backend
- **Секреты**: заменить `Jwt:SecretKey` в `src/LawyerSite.Api/appsettings.json` (и не хранить прод‑секреты в репо).
- **Админ‑доступ**: сменить дефолтный пароль админа (сейчас в `README.md` указан `admin/admin123`).
- **База данных**: поменять имя файла БД (например, `lawyer-site.db`) и строку подключения.
- **CORS/домены**: обновить `Cors:AllowedOrigins` под новый фронтенд‑домен.

### Frontend
- **Название/брендинг**: тексты/цвета/логотипы под адвоката.
- **API base url**: проверить, что прокси (`src/proxy.conf.js`) корректно указывает на новый backend.

## 3) Что можно переиспользовать 1‑в‑1
- **Запись на консультацию** (календарь/слоты) — остаётся практически без изменений.
- **Админ‑панель** — управление расписанием/блокировками/заявками.
- **Блог/контент** — опционально, если адвокату нужна контент‑стратегия.

## 4) Новый функционал — только если необходим (критерии)

Ниже — функции, которые часто нужны адвокату, но **имеет смысл добавлять только при наличии реального запроса**.

### 4.1 Учёт обращений (интейк), если “записи” недостаточно
Нужно, если:\n
- клиент хочет не просто “забронировать слот”, а **описать ситуацию** до звонка/встречи\n
- нужно сортировать обращения по приоритету/типу (семейные, трудовые, уголовные и т.п.)

Минимальный функционал:
- форма обращения: тема, описание, контакты, предпочтительный канал связи
- статус обращения: `New/InReview/NeedsInfo/Scheduled/Closed`

### 4.2 Документы клиента, если есть обмен файлами
Нужно, если:\n
- на входе часто присылают файлы (договоры, решения суда, переписку)\n
- нужен безопасный доступ (только админ/юрист)

Минимальный функционал:
- загрузка + привязка к обращению\n
- ограничение типов/размера (у вас уже есть `UploadSettings`)\n
- аудит (кто/когда скачал) — добавлять только если важно

### 4.3 Уведомления (email/telegram), если есть операционная нагрузка
Нужно, если:\n
- много переносов/подтверждений\n
- клиенты часто “забывают”

Минимальный функционал:
- подтверждение записи\n
- напоминания за N часов\n
- шаблоны сообщений

## 5) Как не потерять багфиксы между репозиториями (важно)

Если два репозитория будут жить долго, лучший компромисс:\n
- выделить общее ядро (модели/валидацию/часть сервисов) в отдельный пакет/репозиторий\n
- подключать его как Git submodule/nuget/npm package\n

Иначе, через 2–3 месяца обычно начинается “двойная работа” и расхождение поведения.
